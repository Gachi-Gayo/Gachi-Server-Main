# gachi-server 개발 워크플로우

## 역할

Figma는 화면과 흐름의 기준, Linear는 작업 명세와 상태의 기준, GitHub는 코드 리뷰·병합의 기준, Slack은 사용자에게 보내는 알림의 기준이다.

작업 전 최신 Feature list와 해당 사용자 흐름 프레임을 확인하고, Linear 이슈를 만든다. 이슈에는 Figma로 해석한 사용자 목표, 화면 전환·분기·상태, API/도메인 범위, 완료 조건, 제외 범위, Figma 링크를 구체적으로 작성한다.

## 검증과 PR

- 코드 구현은 Red → Green → Refactor 순서를 따른다.
- REST API 변경 시 로컬 Happy case 1개와 의미 있는 edge case를 실제 호출하고, PR 템플릿의 API 테스트 표에 요청·기대 응답·실제 응답·판정을 작성한다.
- REST API 변경이 없으면 PR API 테스트 섹션에 `미실시(REST API 변경 없음)`, 사유와 실행한 테스트를 작성한다.
- 모든 PR 제목과 본문은 한국어로 쓰며, `.github/PULL_REQUEST_TEMPLATE.md`의 모든 섹션을 채운다.
- Draft PR 생성 후 정식 GitHub URL이 열리는지 확인하고 Linear에 연결한다. PR이 `dev`에 병합된 뒤에만 Linear를 Done으로 변경한다.

## Slack 고정 템플릿

### 작업 시작

```md
*🚀 작업 시작* · `JIN-<번호>` · <흐름 이름>

• *Linear:* <https://linear.app/...|JIN-<번호>>
• *브랜치:* `<branch>`
• *작업 범위:* <한 줄 요약>
• *예정 검증:* 단위 테스트[, API 테스트]
```

### 수동 작업 요청

```md
*⚠️ 수동 작업 필요* · `JIN-<번호>` · <흐름>

• *필요한 이유:* <한 줄 설명>
• *해야 할 일:* <순서가 있으면 번호 목록>
• *완료 기준:* <확인 방법>
• *완료 후:* Codex App에서 `설정 완료`라고 알려주세요.
```

### 작업 차단

```md
*🚧 작업 차단* · `JIN-<번호>` · <흐름>

• *막힌 이유:* <한 줄 설명>
• *영향:* <일정 또는 기능 영향>
• *필요한 결정:* <선택지와 권장안>
```

### 리뷰 준비 완료

```md
*✅ 리뷰 준비 완료* · `JIN-<번호>` · <흐름>

• *PR:* <https://github.com/<owner>/<repo>/pull/<number>|Draft PR #<number>>
• *구현 요약:* <한두 문장>
• *검증:* <단위·통합 테스트 결과>
• *API 테스트:* 총 <N>건 · PASS <n> · FAIL <n>
  - REST API 변경이 없으면: `미실시(REST API 변경 없음)` · <실행한 테스트>
• *리뷰 시 확인:* <없으면 `없음`>
```
