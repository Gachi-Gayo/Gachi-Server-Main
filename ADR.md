# Architecture Decision Records

이 문서는 이후 작업자가 이미 합의된 중요한 구조·기술 결정을 확인하고, 같은 논의를 반복하거나 기존 선택을 무심코 되돌리는 일을 막기 위한 기록이다.

## 철학

MVP를 빠르게 제공하되, 이후 기능이 늘어날 영역은 도메인 경계와 API 계약을 명확히 유지한다. 검증된 최소 구현을 우선하고, 단일 사용처를 위한 추상화나 외부 의존성은 필요해질 때만 추가한다.

---

### ADR-001: 도메인 중심의 모듈형 모놀리스와 레이어드 구조

**결정**: Spring Boot 단일 애플리케이션 안에서 `user`, `travel`, `itinerary`, `place`, `review` 도메인을 분리하고, 각 도메인에 `controller → service → repository → entity` 계층을 둔다.

**이유**: 현재 MVP 규모에서는 분산 서비스의 운영 복잡도 없이도 도메인별 변경 범위를 분리하고, 기능 흐름 단위 개발을 지원할 수 있다.

**트레이드오프**: 도메인 간 의존성이 커지면 모놀리스 내부 결합이 생길 수 있으며, 독립 배포·독립 확장은 바로 할 수 없다.

### ADR-002: 일관된 API 응답과 예외 계약

**결정**: 모든 Controller 응답은 `ApiResponse<T>`로 감싸고, 비즈니스 오류는 `GachiException`과 `ErrorCode`를 통해 `GlobalExceptionHandler`에서 처리한다.

**이유**: 클라이언트가 성공·실패 응답을 예측 가능한 형식으로 처리할 수 있고, 도메인 서비스가 HTTP 응답 구성에 직접 의존하지 않는다.

**트레이드오프**: 단순한 엔드포인트도 공통 래퍼와 오류 코드를 따라야 하며, HTTP 표준 응답 본문을 그대로 사용하는 것보다 형식상의 제약이 생긴다.

### ADR-003: 카카오 OAuth2 로그인 후 JWT 기반 API 인증

**결정**: 카카오 OAuth2로 사용자를 식별하고, 로그인 성공 뒤 서버가 발급한 JWT로 보호된 API를 인증한다.

**이유**: 사용자 가입 절차를 줄이면서 서버 API 요청을 세션 상태와 분리해 처리할 수 있다.

**트레이드오프**: 카카오 제공 정보와 OAuth 설정에 의존하며, JWT 만료·폐기·보관 정책을 별도로 관리해야 한다.

## 기록 규칙

- ADR은 구조, 데이터 모델, 인증, 외부 연동, API 계약처럼 이후 구현에 지속적으로 영향을 주는 결정만 기록한다.
- 기존 결정을 바꾸면 기존 항목을 삭제하지 않고, 새 ADR에서 대체 대상과 변경 이유를 적는다.
- 구현 세부사항이나 일회성 버그 수정은 Linear 이슈와 PR에 기록한다.
